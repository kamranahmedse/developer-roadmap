# Component Documentation

Good documentation makes design systems usable. Design Engineers often write documentation that helps both designers understand implementation constraints and developers use components correctly.

**What to Document**:
- **Usage**: When and why to use the component
- **Props/API**: All available options with types and defaults
- **Examples**: Code snippets for common use cases
- **Do's and Don'ts**: Visual examples of correct/incorrect usage
- **Accessibility**: Keyboard behavior, ARIA requirements
- **Design Guidelines**: Spacing, sizing, variant usage

**Documentation Tools**:
- **Storybook**: Industry standard for component docs
- **MDX**: Markdown + JSX for rich documentation
- **Docusaurus**: Full documentation sites
- **README files**: Per-component documentation

**Writing Tips**:
- Lead with the most common use case
- Show, don't just tell (live examples)
- Keep code examples minimal and focused
- Update docs when components change
- Include accessibility information

**Storybook Best Practices**:
- One story per meaningful state
- Use Controls for interactive exploration
- Add Docs page with usage guidelines
- Include accessibility addon for testing
